# 즉시 구매 설계 (2026-07-17)

사용자 피드백: "자동 예약구매만 되는 것 같은데, 이번주 로또를 못 샀을 때 바로 즉시 구매하는
기능도 있으면 좋겠다." — 예약을 안 해뒀거나 놓친 사용자를 위한 원탭 구매.

## 정책 (확정)

1. **첫 구매 + 추가 구매** — 이번 회차 미구매(`lastPurchasedRound < 현재 회차`)면 저장 슬롯
   그대로 구매. 이미 구매한 회차에서는 버튼이 **"추가 구매"**로 바뀌어 자동 게임 수(1~5)를
   골라 추가 구매할 수 있다(예: 자동구매가 1게임만 샀을 때 남은 주간 한도 4게임 소진).
   추가분은 **자동 게임만** — 저장된 수동 번호를 또 사면 같은 번호 중복 구매가 되기 때문.
   **주간 5게임 한도 판정은 서버에 위임** — 초과 시 동행복권 에러 메시지("주간구매금액 초과")를
   그대로 표시하고, 앱은 로컬 구매수를 추적하지 않는다(앱 밖 웹사이트 구매는 어차피 못 잡음).
2. **번호는 저장된 슬롯 그대로** — 번호 탭에서 설정한 5슬롯(자동/수동)을 그대로 즉시 실행.
   별도 게임 수 선택 UI 없음. 판정식: **게임 수 = 비-null 슬롯 수** — 부분 설정도 그대로
   구매(예: 2슬롯 설정 = 2게임 ₩2,000, `PurchaseService`의 1~5게임 검증과 정합).
   0게임일 때만 "미설정"으로 보고 번호 설정 화면으로 유도.
3. **자동구매 ON/OFF와 무관** — 로그인 + 슬롯 설정만 있으면 구매 가능. 즉시 구매는 예약의
   보완이 아니라 독립 기능.

## UX

위치: **번호 탭, 설정 저장 버튼 바로 아래**(네이비 CtaButton) — "설정 → 저장 → 바로 구매"
흐름. 홈 화면은 변경하지 않는다. (2026-07-17 오너 피드백: 홈 배치는 부자연스러움 —
초기 설계의 홈 프라이머리 CTA에서 변경.) 미저장 변경이 있으면 구매 대신 "먼저 설정·저장"
스낵바로 유도 — 화면 표시와 실제 구매(저장된 슬롯)의 불일치 방지.

버튼 상태 (위에서부터 우선):

| 상태 | 표시 |
|---|---|
| 비로그인 | 비활성 + "로그인 후 이용 가능" |
| 판매시간 외 (매일 00~06시, 토 20시~일 06시, KST) | 비활성 + "지금은 판매시간이 아닙니다" |
| 이번 회차 구매 완료 | 활성 — "추가 구매" (자동 게임 수 선택 플로우) |
| 저장된 게임 0개 또는 미저장 변경 | 활성 — 탭 시 "먼저 설정·저장" 스낵바 |
| 정상 | 활성 — "지금 바로 구매" |

첫 구매 플로우: 탭 → 확인 다이얼로그("제 N회 · M게임 · ₩M,000을 지금 구매할까요?" — 실결제라
원탭 즉시 결제 금지) → 진행 스피너 → 성공 시 결과 다이얼로그(구매 번호를 LottoBall로 표시) +
잔액 자동 갱신. 이후 같은 회차 예약구매는 기존 가드가 자동 스킵.

추가 구매 플로우: 탭 → 자동 게임 수 선택 다이얼로그(1~5, 금액 표시, 그 자체가 확인 다이얼로그)
→ 진행 스피너 → 결과/에러는 첫 구매와 동일. 한도 초과면 서버 에러 메시지 그대로.

실패는 2분류로 표시한다(아래 동시성·결제 안전 섹션). **자동 재시도는 없음** —
AutoPurchaseWorker와 같은 이유(서버가 이미 처리했을 수 있어 재전송은 중복 결제 위험).

세션: 탭 시점에 앱 세션이 만료됐을 수 있으므로 구매 직전 저장된 자격증명으로 **재로그인 후
구매**(워커와 동일 패턴). 세션 만료 실패 원천 제거, 비용은 1~2초.

## 동시성·결제 안전

- **구매 직렬화**: 예약 워커와 즉시 구매는 같은 앱 프로세스에서 돈다(WorkManager 기본).
  프로세스 전역 구매 Mutex를 하나 두고, 즉시 구매와 워커의 "가드 판정~구매 실행~**회차
  기록 완료**" 구간을 같은 Mutex로 감싼다 — 기록 전에 잠금을 풀면 그 틈에 상대가 이전
  값을 읽는다. 잔액 갱신·알림은 잠금 밖. (앱 프로세스가 없는 워커 단독 실행은 경쟁 상대
  자체가 없다.)
- **Mutex 내 재판정은 모드별**: 즉시 구매는 의도(첫 구매/추가 구매)를 명시적으로 전달한다.
  **첫 구매** — Mutex 획득 후 `lastPurchasedRound`를 SecureStore에서 다시 읽어 이미 구매된
  회차면 구매 없이 중단하되, 에러가 아니라 "방금 이 회차가 구매되었습니다"로 안내하고 CTA를
  추가 구매로 갱신한다(다이얼로그 체류 중 워커 선점 케이스). **추가 구매** — 판매시간·회차만
  재검증하고 `lastPurchasedRound`로 막지 않는다(구매 완료가 정상 전제, 방어선은 서버 한도).
  워커는 기존 가드 그대로(첫 구매 의미론).
- **확정 시점 재검증**: 확인/게임수 다이얼로그가 열린 채 시간이 흐를 수 있다(자정·토 20시
  경계, 그 사이 워커 구매 완료 등). 확인 버튼 시점에 판매시간·회차·가드를 다시 검증하고,
  무효면 구매 요청 없이 다이얼로그를 닫고 사유를 표시한다.
- **실패 2분류**: ① **서버 확정 거절**(응답에 실패 메시지 — 주간구매금액 초과, 잔액 부족 등):
  메시지를 그대로 표시, 재시도해도 안전. ② **결과 불명**(네트워크 오류·타임아웃 — 서버가
  처리했는지 알 수 없음): "구매가 접수됐을 수 있습니다. 내역 탭에서 확인 후 다시
  시도해주세요"로 안내하고 재시도를 유도하지 않는다. 회차 가드는 **앱이 성공 응답을 관측한
  경우에만** 기록되므로 "재시도해도 가드가 막아준다"는 보증은 성립하지 않으며(추가 구매는
  애초에 가드 대상이 아님), 결과 불명에서의 방어선은 이 안내 문구다.
- **성공 커밋 순서**: 성공 응답 수신 즉시 `setLastPurchasedRound` 기록이 최우선. 이후 잔액
  갱신 실패는 성공 표시에 영향을 주지 않는다(성공 다이얼로그 그대로, 잔액만 이전 값 유지).
- **잔여 리스크(수용)**: 성공 응답 수신~회차 기록 사이 프로세스 킬 창은 워커와 동일하게
  남는다. 요청 전송 후(InProgress) 프로세스 킬·화면 이탈로 앱이 결과를 못 본 경우도 같은
  성격 — 재진입 시 안내 없이 Idle로 시작한다. 완전 차단(영속 pending 마커 + 서버 구매내역
  대조)은 과설계로 도입하지 않는다(워커의 기존 수용 결정과 동일). 결과 불명 안내가 완화책.

## 구현 (변경 6곳, 새 파일 없음)

1. **AppContainer.kt** — `suspend fun instantPurchase(extra: Boolean, autoGames: Int, manualNumbers): PurchaseResult?`:
   구매 Mutex 안에서 모드별 재판정(동시성 섹션 — 첫 구매만 가드, 차단 시 null 반환 = "이미
   구매됨" 신호) → 자격증명 읽기 → `auth.login()` → `PurchaseService.purchase()` → 성공 즉시
   `setLastPurchasedRound()` + 구매 계정 기록 — 여기까지 임계구역. Mutex 해제 후
   `refreshBalance()`(잔액부족 알림 포함, 워커와 동일 — 실패해도 성공 표시 유지).
   `lastPurchasedRound`는 StateFlow로 노출하고 스플래시 하이드레이션·구매 성공·홈 새로고침 때
   갱신(워커가 백그라운드에서 갱신한 경우 대비). **회차 가드의 계정 스코프**: 구매 기록 시
   구매 계정 ID를 병기하고, 수동 로그인 시 계정이 기록과 다르면 `lastPurchasedRound`를 0으로
   리셋 — 계정 전환 오판은 막고 같은 계정 재로그인 기록은 보존한다(로그아웃 시 초기화하지
   않음 — 재로그인 후 수동 번호 중복 구매 방지). 이 값은 이 기기·이 앱의 기록일 뿐 웹 구매는
   반영하지 못한다(수용, 서버 한도가 상한 방어).
2. **ViewModels.kt (NumberViewModel)** — `InstantState`(Idle/NeedsSetup/Confirming/
   PickingExtra/InProgress/Success/AlreadyPurchased/SaleClosed/RoundChanged/Error)
   StateFlow + 탭·확인·닫기 액션. 첫 구매 탭은 저장된 슬롯을 읽어 Confirming으로, 구매완료
   회차 탭은 PickingExtra(게임 수 선택)로 분기. 판매시간·구매완료 게이트는 **탭 시점과
   확정(확인 버튼) 시점에 각각 재검증**(표시용 계산은 구성 시점이라 자정 넘김 등에서
   stale 가능 — 동시성 섹션). 판매시간 판정은 회차 계산과 동일하게 KST 고정,
   `SettingsViewModel.isValidPurchaseTime` 재사용.
3. **NumberScreen.kt** — 저장 버튼 아래 즉시구매/추가구매 CTA, 확인·게임수선택·결과·에러
   다이얼로그, 상태별 비활성 라벨. 게임 0개·미저장 변경은 기존 스낵바로 안내(HomeScreen
   무변경).
4. **strings.xml ko/en/ja** — 버튼·다이얼로그·상태 라벨 ~12개 × 3언어.
5. **AutoPurchaseWorker.kt** — "가드 판정~구매 실행~회차 기록"을 공유 구매 Mutex로 감싸고
   구매 계정 기록 1줄 추가하는 최소 수정만(동시성 섹션). 그 외 로직 무변경.
6. **SecureStore.kt** — 구매 계정 키 1개(`last_purchase_owner`) 게터/세터 추가.

건드리지 않는 것: PurchaseService, 알람 스케줄러.
즉시 구매가 남긴 `lastPurchasedRound`를 워커 가드가 읽으므로 예약구매 중복 방지는 공짜.

## 검증

- 컴파일 + `./gradlew test`(기존 단위테스트 회귀) + assembleDebug.
- 에뮬레이터 버튼 상태: 비활성 2종(비로그인 / 판매시간 외) 라벨, 활성 3분기(미구매 회차 →
  "지금 바로 구매" / 구매완료 회차 → "추가 구매" 전환 / 0게임 → 번호 설정 유도).
- 다이얼로그 플로우: 확인 취소, 게임수 선택 취소, 확정 시점 재검증(기기 시간 변조로 무효화),
  프로세스 재생성 후 확인 다이얼로그가 복원되지 않고 Idle로 시작하는지.
- 실서버 필요 항목(실결제 성공, 주간 한도 초과 메시지, 잔액 부족, 결과 불명)은 에뮬레이터로
  재현 불가 — 배포 전 실계정 수동 시나리오(첫 구매 1게임 ₩1,000 → 같은 회차 추가 구매 →
  내역 탭 확인)는 사용자와 별도 협의.
